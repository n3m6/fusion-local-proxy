## Findings for Q13

### Summary
Three widely-used structured-logging libraries exist for Node.js/TypeScript: **pino** (v10.3.1), **winston** (v3.19.0), and **bunyan** (v1.8.15). Each provides APIs for attaching structured metadata to log entries, serializing error objects with type/code and message, and measuring or recording per-operation latency — though the mechanisms differ across libraries.

### Details

#### 1. pino (v10.3.1) — `pino`
**Attaching structured metadata:**
- **Per-call metadata**: the first argument to any log method (`logger.info({ key: val }, 'message')`) is a *mergingObject* whose enumerable keys are copied into the JSON log line (API docs: `mergingObject` parameter).
- **Persistent metadata**: `logger.child({ key: val })` returns a child logger that includes the given bindings on every log line (`docs/api.md`, `logger.child(bindings)` section).
- **Dynamic per-log metadata**: the `mixin()` constructor option is called on every log invocation, returning an object merged into each log line (`docs/api.md`, `mixin` option, line ~149–171).
- **Base metadata**: the `base` option defaults to `{ pid, hostname }` and can be overridden or set to `undefined` (`docs/api.md`, `base` option, line ~489–495).

**Measuring latency:**
- Pino has no built-in profiler. Latency must be measured externally (e.g., `Date.now()` / `process.hrtime()`) and the computed duration attached as a property on the *mergingObject*:
  ```js
  const start = Date.now();
  // ...operation...
  logger.info({ latency_ms: Date.now() - start }, 'operation complete');
  ```
  (Inferred from the mergingObject API — no dedicated `profile`/`startTimer` method exists in pino's API surface.)

**Recording error details (codes, truncated messages):**
- **Default error serialization**: when an `Error` instance is passed as the first argument, it is serialized with fields `type` (the error class name, e.g. `"Error"`), `msg` (the error's `.message`), and `stack` (`docs/api.md`, `error-serialization` section, lines ~844–849).
- **Custom error serialization**: the `serializers` option accepts an object mapping property names to serializer functions. The default is `{ err: pino.stdSerializers.err }`. A custom `err` serializer can extract an error code property and truncate the message (`docs/api.md`, `serializers` option, lines ~454–468).
- **Error key**: the `errorKey` option renames the error property from `'err'` to another string (`docs/api.md`, `errorKey` option, line ~522).
- The `pino.stdSerializers` object (re-exported from `pino-std-serializers`) provides `err`, `req`, `res`, and wrapper helpers (`docs/api.md`, `pino.stdSerializers` section, lines ~1460–1462).
- **Message truncation**: not built-in; achieved by implementing a custom serializer or using the `hooks.logMethod` hook to mutate arguments before serialization (`docs/api.md`, `hooks.logMethod` section).

**npm registry data** (`https://registry.npmjs.org/pino/latest`): version 10.3.1, description "super fast, all natural json logger," ~195 files, dependencies include `pino-std-serializers`, `atomic-sleep`, `thread-stream`.

#### 2. winston (v3.19.0) — `winston`
**Attaching structured metadata:**
- **Persistent metadata**: the `defaultMeta` option on `createLogger()` adds the same metadata to every log (`winston` README, `createLogger` example, lines ~50–55).
- **Child loggers**: `logger.child({ requestId: '451' })` creates a child logger with bound metadata (`winston` README, "Creating child loggers" section).
- **Per-call metadata**: any properties beyond `level` and `message` on the `info` object are treated as metadata. The canonical form is `logger.log({ level: 'info', message: 'text', myMeta: 42 })` or convenience methods like `logger.info('message', { myMeta: 42 })` (`winston` README, "Streams, `objectMode`, and `info` objects" section).

**Measuring latency:**
- **`logger.profile(id)`**: call with the same string `id` twice to start and stop; winston emits an info-level log with `durationMs` calculated automatically (`winston` README, "Profiling" section, lines ~969–1006).
- **`logger.startTimer()`**: returns a profiler object; call `.done({ message: '...' })` to log the elapsed duration. Metadata can be attached via the object passed to `.done()` (`winston` README, "Profiling" section, lines ~992–1000).
- Profile log level defaults to `'info'` but can be overridden by passing `{ level: 'debug' }` metadata (`winston` README, line ~1002–1006).

**Recording error details:**
- **Uncaught exceptions/rejections**: `logger.exceptions.handle()` and `logger.rejections.handle()` capture process-level uncaught errors (`winston` README, "Exceptions" and "Rejections" sections).
- **Per-log errors**: errors are handled as ordinary metadata. The developer passes error information as properties on the `info` object; custom formats can transform them. There is no default `err` serializer equivalent to pino/bunyan — error serialization is user-defined via `logform` formats (`winston` README, "Creating custom formats" section).
- **`exitOnError`**: controls whether handled exceptions cause `process.exit` (default: `true`) (`winston` README, `createLogger` table, line ~164).

**npm registry data** (`https://registry.npmjs.org/winston/latest`): version 3.19.0, description "A logger for just about everything," dependencies include `logform`, `winston-transport`, `triple-beam`.

#### 3. bunyan (v1.8.15) — `bunyan`
**Attaching structured metadata:**
- **Persistent metadata**: any extra fields passed to `bunyan.createLogger()` beyond the named config keys are added to every log record (`bunyan` README, "Constructor API" section, lines ~155–161).
- **Child loggers**: `log.child({ widget_type: 'wuzzle' })` creates a child logger with bindings included in every log record (`bunyan` README, "log.child" section, lines ~280–340).
- **Per-call metadata**: the first argument to a log method can be a fields object merged into the log record: `log.info({ foo: 'bar' }, 'message')` (`bunyan` README, "Log Method API" section, lines ~178–180).
- **Best-practice recommendation**: always give a field name to included objects, e.g. `log.info({ widget: mywidget })` rather than `log.info(mywidget)`, to avoid key collisions with core fields (`bunyan` README, lines ~200–210).

**Measuring latency:**
- No built-in profiler. The bunyan documentation suggests a `latency` custom field as a "field to consider" for HTTP response latency (`bunyan` README, "Other fields to consider" section, line ~735). The pattern is manual measurement and attachment:
  ```js
  log.info({ latency: Date.now() - start }, 'request complete');
  ```

**Recording error details (codes, truncated messages):**
- **Default error logging**: `log.info(err)` serializes the error with fields: `message` (the error message), `name` (the error class name, e.g. `"TypeError"`), and `stack` (`bunyan` README, "Recommended/Best Practice Fields" section, lines ~678–694).
- **Standard error serializer**: `bunyan.stdSerializers.err` traverses the error's cause chain (for `verror`-style wrapped errors) (`bunyan` README, "Standard Serializers" table, lines ~490–496).
- **Usage with custom fields**: `log.error({ err: err, otherField: val }, 'message')` — requires the `err` serializer to be registered on the logger (`bunyan` README, lines ~189–198).
- **Request correlation**: `req_id` is a recommended field for collating all log entries related to a single request (`bunyan` README, lines ~696–702).

**npm registry data** (`https://registry.npmjs.org/bunyan/latest`): version 1.8.15, description "a JSON logging library for node.js services," optional dependencies include `dtrace-provider`, `mv`, `moment`, `safe-json-stringify`.

#### Common Pattern: Per-Operation Structured Logging with Latency and Error Metadata

Across all three libraries, the idiomatic pattern for per-operation structured logging is:

1. **Create a logger instance** with application-level metadata (`name`, `service`, etc.).
2. **Derive a child logger** (or use `defaultMeta`/base bindings) with operation-scoped metadata (e.g., `operation: 'judge-step'`, `requestId`).
3. **Measure latency** by capturing a timestamp before the operation, computing the delta afterward.
4. **On success**: log at `info` level with the latency value and any result metadata as structured fields on the log object.
5. **On error**: log at `error`/`warn` level, passing the error object. The library's error serializer (pino/bunyan) will produce `type`/`name` (as error code), `msg`/`message` (the error message), and `stack`. For truncated messages, a custom serializer or format function truncates `error.message` to a maximum length before logging.

**Example in pino** (illustrative of the pattern, not a recommendation):
```js
const logger = pino({ name: 'fusion-proxy' });
const opLog = logger.child({ stage: 'judge' });

const start = Date.now();
try {
  // ...operation...
  opLog.info({ latency_ms: Date.now() - start, cost: 0.002 }, 'stage complete');
} catch (err) {
  opLog.error({
    latency_ms: Date.now() - start,
    err,                          // serialized as { type, msg, stack }
    failed_model: {
      identifier: err.modelId,
      error_code: err.code,
      truncated_message: err.message.slice(0, 200)
    }
  }, 'stage failed');
}
```

**Example in winston** (illustrative, not a recommendation):
```js
const logger = winston.createLogger({ defaultMeta: { service: 'fusion-proxy' } });
const profiler = logger.startTimer();
try {
  // ...operation...
  profiler.done({ message: 'stage complete', stage: 'judge', cost: 0.002 });
} catch (err) {
  profiler.done({
    level: 'error',
    message: 'stage failed',
    stage: 'judge',
    failed_model: {
      identifier: err.modelId,
      error_code: err.code,
      truncated_message: err.message.slice(0, 200)
    }
  });
}
```
