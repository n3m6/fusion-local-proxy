#### 1. `Promise.allSettled` result shapes (fulfilled vs. rejected)

`Promise.allSettled` takes an iterable of promises and returns a single `Promise` that **always resolves** — it never rejects. The resolved value is an array of **settlement objects**, one per input promise, preserving input order. Each settlement object is a plain object with exactly one discriminator property:

- **`status`**: `"fulfilled"` — the promise resolved.
  - The object also has a **`value`** property containing the resolution value.
  - Shape: `{ status: "fulfilled", value: <resolution> }`

- **`status`**: `"rejected"` — the promise rejected.
  - The object also has a **`reason`** property containing the rejection reason (typically an `Error` instance, but can be any value thrown/rejected).
  - Shape: `{ status: "rejected", reason: <error> }`

**TypeScript type** (from `lib.es2020.promise.d.ts`, shipped with TypeScript 3.9+):
```ts
interface PromiseFulfilledResult<T> {
  status: "fulfilled";
  value: T;
}
interface PromiseRejectedResult {
  status: "rejected";
  reason: any;
}
type PromiseSettledResult<T> = PromiseFulfilledResult<T> | PromiseRejectedResult;
```

**Discrimination pattern** — callers iterate the array and branch on `status`:
```ts
const results = await Promise.allSettled(promises);
for (const r of results) {
  if (r.status === "fulfilled") {
    console.log("Success:", r.value);
  } else {
    console.error("Failure:", r.reason);
  }
}
```

MDN documents this at: `Promise.allSettled()` — return value is “an array of objects, each describing the outcome of each promise … with a `status` string and, depending on the status, a `value` or `reason`.”

#### 2. Error-aggregation pattern A — Collect-and-classify with partial-success envelope

This pattern processes the `Promise.allSettled` array into a **structured result envelope** that separates successes from failures, allowing the caller to distinguish partial failure (some succeeded, some failed) from total failure (none succeeded).

**Mechanics**:
1. Partition the settlement array by `status`.
2. Extract `value` from fulfilled entries and `reason` from rejected entries.
3. Normalise each `reason` into a lightweight error descriptor (identifier, code, message) rather than leaking raw Error objects.
4. Wrap everything in a domain result type that carries **succeeded results**, a **failed-models array**, and a derived flag or method for total-failure checks.

**Conceptual implementation sketch** (matching FR-4 / AC-7 shapes from the specification):
```ts
interface ModelResult { modelId: string; output: string; }
interface FailedModel { modelId: string; errorCode: string; message: string; }

interface PanelResult {
  succeeded: ModelResult[];
  failed: FailedModel[];
  get allFailed(): boolean;  // true when succeeded.length === 0
}

async function runPanel(models: ModelRef[], call: (m: ModelRef) => Promise<string>): Promise<PanelResult> {
  const settled = await Promise.allSettled(
    models.map(m => call(m).then(output => ({ modelId: m.id, output })))
  );

  const succeeded: ModelResult[] = [];
  const failed: FailedModel[] = [];

  for (const r of settled) {
    if (r.status === "fulfilled") {
      succeeded.push(r.value);
    } else {
      const err = r.reason instanceof Error ? r.reason : new Error(String(r.reason));
      failed.push({
        modelId: /* extract from context or closure */,
        errorCode: (err as any).code ?? "UNKNOWN",
        message: err.message.slice(0, 200),  // truncated per NFR-5
      });
    }
  }

  return { succeeded, failed, get allFailed() { return succeeded.length === 0; } };
}
```

**When to use**: When the caller needs both partial results and diagnostic failure information, and the system should continue with partial success unless *every* promise failed. This is the pattern the specification labels “partial failures non-fatal; `all_panels_failed` fatal.”

This pattern is recommended in Node.js error-handling best practices (e.g., “Handle operational errors separately; never throw inside `Promise.allSettled` processing — classify and report”).

#### 3. Error-aggregation pattern B — `AggregateError` (native, ES2021 / Node.js 15+)

`AggregateError` is a built-in `Error` subclass that represents **multiple errors wrapped in a single error object**. It is the native language construct for “all of these operations failed.”

**Constructor**:
```ts
new AggregateError(errors: Iterable<any>, message?: string);
```

**Properties**:
- `name`: `"AggregateError"`
- `message`: the supplied message string (or empty)
- **`errors`**: an array of the individual errors passed to the constructor (the iteration order is preserved).

**Integration with `Promise.allSettled`**:

```ts
const settled = await Promise.allSettled(promises);
const failures = settled
  .filter((r): r is PromiseRejectedResult => r.status === "rejected")
  .map(r => r.reason);

if (failures.length > 0) {
  throw new AggregateError(failures, `All ${promises.length} panel calls failed`);
}
```

`AggregateError` is also thrown by `Promise.any()` when **every** input promise rejects — the same “total failure” semantics required by the specification (`all_panels_failed`). The `errors` property on `AggregateError` can carry rich per-model error descriptors if the caller normalises each rejection into a structured object before passing it.

**When to use**: When total failure should be surfaced as a single throwable error that still preserves individual failure causes. This maps directly to the `all_panels_failed` `FusionError` described in the specification (FR-4, NFR-5, AC-7). `FusionError` in the domain model can either wrap or extend `AggregateError`, or adopt the same `errors[]` shape.

#### Pattern comparison

| Aspect | Pattern A (envelope) | Pattern B (AggregateError) |
|--------|---------------------|---------------------------|
| Partial success handling | First-class — successes and failures coexist in one return value | Not supported — only thrown on total failure |
| Total failure signalling | `result.allFailed` flag or `succeeded.length === 0` | `throw new AggregateError(...)` |
| Caller experience | Always returns; caller checks `allFailed` or `failed.length` | Caller wraps in try/catch; only catches on total failure |
| Composable with other patterns | Yes — the envelope can internally use `AggregateError` for the failure payload | Yes — can be caught and converted into an envelope result |
| Native Node.js support | N/A (application-level pattern) | Built-in since Node.js 15.0.0 / ES2021 |

These two patterns are not mutually exclusive. A common production approach **combines** them: the `PanelRunner` returns an envelope (Pattern A) for partial failures, but when `allFailed` is true, it throws an `AggregateError` (or a domain `FusionError` extending it) so that the orchestrating use case can cleanly abort the pipeline (Pattern B for total failure).
