## Findings for Q9

### Summary
The OpenAI chat completions API returns token usage via `CompletionUsage` (with `prompt_tokens`, `completion_tokens`, `total_tokens`, and optional detail breakdowns); timing is limited to a Unix `created` timestamp. The Anthropic Messages API returns token usage via `Usage` (with `input_tokens`, `output_tokens`, cache-token breakdowns, `thinking_tokens`, server-tool-use counts, `service_tier`, and `inference_geo`); it returns **no** latency, duration, or timestamp fields. Neither API provides server-side processing-time or latency measurements in the chat/messages response body. The types were extracted from installed packages: `openai@6.42.0` and `@anthropic-ai/sdk@0.104.1`.

#### 2. OpenAI Chat Completions — Timing / Latency Fields

**There are no latency or duration fields returned in the OpenAI chat completions response.**

The only time-related field is:

| Field path | Type | Description |
|---|---|---|
| `created` | `number` | Unix timestamp (in seconds) of when the completion was created. Present on both `ChatCompletion` (`completions.d.ts:169`) and `ChatCompletionChunk` (`completions.d.ts:568`). |

No `response_ms`, `processing_time`, `latency`, or similar field exists in any chat completions response or chunk type. Client-side latency must be measured by the caller using wall-clock timing around the SDK call.

#### 4. Anthropic Messages API — Timing / Latency Fields

Client-side latency must be measured by the caller.
