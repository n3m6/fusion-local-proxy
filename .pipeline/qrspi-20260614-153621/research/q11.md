#### Stubbing/Mocking Patterns for Port Interfaces
The vitest `@vitest/spy` package exposes the core mocking primitives (`/home/n3m6/src/zoink/zoink-cli/node_modules/@vitest/spy/dist/index.d.ts`):

- **`vi.fn<F>(implementation?: F): Mock<F>`** — creates a type-safe mock function. Without an implementation, returns `undefined`; with one, delegates to it. Returns a `MockInstance` with full call-tracking (`.mock.calls`, `.mock.results`, `.mock.lastCall`) and behavior-control methods (lines 197–200 in the type definition).
- **`MockInstance` methods** for stubbing return values (lines 140–242):
  - `.mockImplementation(fn)` — permanent replacement implementation
  - `.mockImplementationOnce(fn)` — one-shot override
  - `.mockReturnValue(value)` / `.mockReturnValueOnce(value)` — synchronous stubs
  - `.mockResolvedValue(value)` / `.mockResolvedValueOnce(value)` — promise-resolving stubs (key for async port methods)
  - `.mockRejectedValue(error)` / `.mockRejectedValueOnce(error)` — promise-rejecting stubs
  - `.mockClear()`, `.mockReset()`, `.mockRestore()` — lifecycle management
- **`vi.spyOn(obj, method)`** — spies on existing object methods; returns `Mock<T>` — `/home/n3m6/src/zoink/zoink-cli/node_modules/@vitest/spy/dist/index.d.ts:277–288`.
- **`vi.mock(path, factory?)`** — hoisted module mocking; factory receives `importOriginal` helper for partial mocking — `/home/n3m6/src/zoink/zoink-cli/node_modules/@vitest/mocker/dist/types.d-BjI5eAwu.d.ts:91` defines `ModuleMockFactoryWithHelper<M> = (importOriginal: () => Promise<T>) => Awaitable<Partial<M>>`.
- **Pattern for hexagonal port stubbing**: Create a plain object satisfying the port interface, with each method assigned a `vi.fn()` configured via `.mockResolvedValue()` or `.mockImplementation()`. The sibling project demonstrates this in `/home/n3m6/src/zoink/zoink-cli/apps/cli/src/adapters/DockerImageManagerAdapter.spec.ts:9–13`:
  ```ts
  const mockInspect = vi.fn();
  const mockPull = vi.fn();
  const mockDocker = {
    getImage: vi.fn(() => ({ inspect: mockInspect })),
    pull: mockPull,
    // ...
  };
  ```
  And `.mockResolvedValue()` / `.mockRejectedValue()` for async behavior (lines 48–62 of the same file).
- **Lifecycle**: `vi.clearAllMocks()` in `beforeEach` resets call tracking between tests — `/home/n3m6/src/zoink/zoink-cli/apps/cli/src/adapters/DockerImageManagerAdapter.spec.ts:41–43`.
- For domain-layer tests (AC-14), **no mocks are needed** — pure functions are tested directly with `expect()` assertions. Demonstrated in `/home/n3m6/src/zoink/zoink-cli/packages/shared/src/severity.spec.ts` (pure `detectSeverity` function tested with `expect().toBe()`).
- For application-layer tests (AC-14), stubbed `ChatModelPort` and `ConfigPort` implementations are passed to the use case constructor or method, allowing orchestration logic to be validated without real SDKs — pattern described in `goals.md:83`.
