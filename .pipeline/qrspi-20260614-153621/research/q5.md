#### 1. Directory Layout Conventions

This follows the established hexagonal convention of grouping by architectural layer (not by feature), with ports defined in the layer that owns them: outbound ports in `domain/ports/` (driven side), inbound ports in `application/ports/` (driving side). The literature (Cockburn, Vernon, Hombergs) consistently places outbound port interfaces in the domain/application boundary and inbound ports in the application layer, exactly as this project plans.

#### 4. Import Rule Enforcement Techniques

The project's acceptance criteria AC-2 (`goals.md:71`) requires verification that `src/domain/` has zero imports from `src/application/` or `src/infrastructure/`, and `src/application/` has zero imports from `src/infrastructure/`. However, the planning documents do **not** specify a concrete enforcement mechanism. Three established techniques exist:

**Technique 1: `eslint-plugin-import-boundaries` (or `eslint-plugin-boundaries`)**
- An ESLint plugin that enforces import rules based on directory structure
- Configured via `.eslintrc` with `boundaries/elements` defining each layer and `boundaries/rules` specifying allowed/disallowed import paths
- Example rule: `{ from: 'src/domain', disallow: ['src/application', 'src/infrastructure'] }`
- Runs at lint time; CI-friendly, blocks violating imports before they merge

**Technique 2: `dependency-cruiser`**
- A standalone tool (can run as CLI or integrated into build scripts) that traverses the module graph and validates rules defined in `.dependency-cruiser.js`
- Rule format: `{ from: { path: '^src/domain' }, to: { path: '^src/application|^src/infrastructure' }, rules: { forbid: true } }`
- Generates visual dependency graphs (useful for architecture docs)
- Can run as a CI check; more powerful than eslint for cross-cutting architecture rules

**Technique 3: TypeScript `paths` aliases + `no-restricted-imports`**
- `tsconfig.json` `paths` can create layer aliases (e.g., `@domain/*`, `@application/*`)
- Combined with ESLint's `no-restricted-imports` rule to forbid relative imports crossing layer boundaries
- Less robust than dedicated architecture linting tools but requires no additional dependencies
- Relies on discipline around alias usage; easy to circumvent with relative paths unless combined with a path-restriction rule

**Technique 4: `@nrwl/nx` module boundaries (if using Nx monorepo tooling)**
- Nx's `@nx/enforce-module-boundaries` rule with `tags` in `project.json` for each library
- Not applicable to single-package projects but common in enterprise hexagonal TypeScript setups

The project's current stance (`requirements.md:75` — "Each phase respects the dependency rule") implies manual verification during code review, with AC-2 (`goals.md:71`) as the acceptance gate, but no automated enforcement mechanism has been selected.
