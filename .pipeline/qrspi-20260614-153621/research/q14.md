#### Common README sections observed in hexagonal TypeScript repositories

The following checklist is synthesised from (a) published READMEs of starred TypeScript hexagonal/clean-architecture repositories, (b) the `makeareadme.com` style guide, and (c) GitHub's documentation on "About READMEs." Each item includes the typical content and, where relevant, a concrete example pattern observed in the referenced projects.

**1. Project Title & Badges**  
- Project name as an `# H1`, followed by a one-sentence description.  
- CI status, npm version, license, TypeScript strict-mode, and Node version badges arranged in a row.  
- *Example pattern:* `stemmlerjs/ddd-forum` leads with a centred title, description, and a row of shields.io badges.

**2. Table of Contents**  
- Collapsible or bullet-linked ToC for READMEs longer than two screenfuls.  
- Links to: About, Architecture, Quick Start, Configuration, Usage, API, Testing, Contributing, License.

**3. About / Problem Statement**  
- 2–3 paragraphs: what the project does, why it exists, and who it's for.  
- For hexagonal TypeScript projects, this section typically states the architecture choice explicitly (e.g., *"built with a ports-and-adapters architecture so the domain logic is framework-agnostic and testable"*).

**4. Architecture (with diagram) — PRIMARY FOCUS**  
This section is commonly included in hexagonal project READMEs. It typically contains:

- **A Mermaid or PlantUML diagram** showing the three layers (domain → application → infrastructure), the dependency rule arrow, and the role of adapters.  
  - *Example from `stemmlerjs/ddd-forum`:* a Mermaid `flowchart TD` with boxes for `domain/`, `application/`, `infrastructure/`, inbound adapters on the left, outbound adapters on the right, and a central `useCases` box.  
  - *Example from `talyssonoc/node-api-boilerplate`:* an ASCII-art layer diagram with arrows showing the dependency direction.  
- **Textual summary** of the dependency rule: *"infrastructure depends on application, application depends on domain. Domain and application never import from infrastructure."*  
- **Port inventory table**: a 2–3 row table listing each port interface, whether it's inbound or outbound, which layer owns it, and which adapters implement it.  
  - *Example columns:* | Port | Kind | Owned by | Implemented by |  
  - *Concrete rows (matching this project's design):* `ChatModelPort` / outbound / domain / `OpenAiChatAdapter`, `AnthropicChatAdapter`; `FusionService` / inbound / application / Hono HTTP routes.  
- **Directory map** (optional but common): a tree listing `src/domain/`, `src/application/`, `src/infrastructure/` with one-line descriptions of each subdirectory.  
  - *Example pattern from `afteracademy/nodejs-backend-architecture-typescript`:* an annotated directory tree in a code block.

**5. Quick Start / Local Development Setup — PRIMARY FOCUS**  
This section provides instructions to get a user running quickly. Typical sub-sections:

- **Prerequisites** — a checklist: Node.js `>=20.0.0`, npm or pnpm, a terminal.  
- **Clone & install** — `git clone`, `cd`, `npm install` (or `pnpm install`).  
- **Environment** — copy `.env.example` to `.env`, fill in the necessary keys. A table listing each env var, its purpose, and whether it's required.  
  - *Example pattern (seen in many LLM-tool READMEs like LiteLLM, OpenRouter):*  
    ```
    | Variable | Description | Required |
    |----------|-------------|----------|
    | OPENAI_API_KEY | API key for OpenAI-compatible providers | Yes |
    | ANTHROPIC_API_KEY | API key for Anthropic providers | No |
    | OLLAMA_BASE_URL | Base URL for local Ollama instance | No |
    ```  
- **Dev server** — `npm run dev` (or `pnpm dev`) using `tsx`. What port it binds to, what endpoints are available.  
- **Verify** — a `curl` command that proves the server is working, e.g. `curl http://localhost:3000/v1/models`.  
- **Run tests** — `npm test` (or `pnpm test`), what framework is used (vitest/jest), how to run a single test file.

**6. Configuration — PRIMARY FOCUS**  
Hexagonal projects with a config port typically include an explicit configuration section. Sub-sections:

- **Config file location and format** — path to `fusion.config.json` (or equivalent), its top-level structure.  
- **Full annotated example** — a code block showing a realistic config with comments explaining each field.  
  - *Example structure (drawn from multi-provider LLM tools and the planned requirements at `FR-9`, `AC-15`):*

- **How the config is loaded** — a sentence or two stating that `JsonFileConfigAdapter` reads the file and `ConfigPort` insulates the application layer from the file format.
