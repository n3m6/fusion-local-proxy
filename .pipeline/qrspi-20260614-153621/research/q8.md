### Details

#### Zod v3: Schema definition for nested objects with typed fields

Zod schemas are built compositionally from primitive and structural constructors.

**Primitive types** — `z.string()`, `z.number()`, `z.boolean()`, `z.bigint()`, `z.date()`, `z.undefined()`, `z.null()`, `z.any()`, `z.unknown()`, `z.void()`.

**Object schemas** — `z.object({ key: schema, ... })` defines a strict object shape. Nested objects are expressed by nesting `z.object()` calls:

```ts
const AnalysisSchema = z.object({
  consensus: z.string(),
  contradictions: z.array(z.object({
    source_a: z.string(),
    source_b: z.string(),
    description: z.string(),
  })),
  unique_insights: z.array(z.object({
    source: z.string(),
    insight: z.string(),
  })),
  blind_spots: z.array(z.string()),
});
```

**Modifiers** — `.optional()`, `.nullable()`, `.nullish()` (nullable + optional), `.default(value)`. For example: `z.string().optional()` makes a field accept `string | undefined`.

**Arrays and records** — `z.array(itemSchema)`, `z.record(keySchema, valueSchema)` (e.g., `z.record(z.string(), z.number())`).

**Unions and discriminated unions** — `z.union([A, B])`, `z.discriminatedUnion("type", [...])`.

**Enums and literals** — `z.enum(["a", "b"])`, `z.literal("value")`.

**Tuples** — `z.tuple([z.string(), z.number()])` for fixed-position arrays.

**Refinements** — `.refine(validator, { message })` and `.superRefine((val, ctx) => { ... })` for custom validation logic.

#### Parse vs. safeParse

Zod provides two primary validation methods:

1. **`schema.parse(data: unknown): T`** — Returns the parsed/validated data on success. **Throws a `ZodError`** if validation fails. This is a "fail-fast" approach suitable when invalid data should stop execution.

   ```ts
   // Throws ZodError if data doesn't match AnalysisSchema
   const analysis = AnalysisSchema.parse(rawJson);
   ```

2. **`schema.safeParse(data: unknown): SafeParseReturnType<T>`** — Returns a **discriminated union result object** that never throws:
   - Success: `{ success: true, data: T }`
   - Failure: `{ success: false, error: ZodError }`

   ```ts
   const result = AnalysisSchema.safeParse(rawJson);
   if (result.success) {
     // result.data is the validated Analysis
   } else {
     // result.error is a ZodError
   }
   ```

   This is the pattern referenced in **AC-8** for graceful degradation: the `JudgeStep` would use `safeParse` to avoid crashing the pipeline on validation failure.

**Async variants** — `schema.parseAsync(data)` and `schema.safeParseAsync(data)` exist for schemas that contain async refinements or transformations.

#### Inferred types

Zod automatically infers the TypeScript type from a schema:
```ts
type Analysis = z.infer<typeof AnalysisSchema>;
// Analysis = { consensus: string; contradictions: { source_a: string; ... }[]; ... }
```

`z.input<typeof schema>` gives the input type (before transformations), while `z.infer<typeof schema>` gives the output type (after transformations/defaults).

#### ZodError structure

When validation fails, `ZodError` provides detailed diagnostics:

| Property / Method | Description |
|---|---|
| `error.issues: ZodIssue[]` | Array of individual validation failures |
| `error.errors: ZodIssue[]` | Alias for `.issues` |
| `error.isEmpty: boolean` | `true` if no issues |
| `error.format(): ZodFormattedError<T>` | Returns a nested object mirroring the schema shape, with `_errors: string[]` at each level |
| `error.flatten(fieldErrors?): { formErrors: string[], fieldErrors: Record<string, string[]> }` | Flattens issues to top-level `formErrors` and dot-path keyed `fieldErrors` |
| `error.toString(): string` | Human-readable summary |
| `error.addIssue(issue): void` | Mutate error (used in `.superRefine()`) |
| `error.addIssues(issues): void` | Mutate error with many issues |

Each **`ZodIssue`** object contains:

| Field | Type | Description |
|---|---|---|
| `code` | `z.ZodIssueCode` | Machine-readable error code: `'invalid_type'`, `'invalid_string'`, `'too_small'`, `'too_big'`, `'invalid_enum_value'`, `'unrecognized_keys'`, `'invalid_union'`, `'invalid_arguments'`, `'invalid_return_type'`, `'custom'`, etc. |
| `path` | `(string \| number)[]` | Path to the failing field (e.g., `['contradictions', 0, 'source_a']`) |
| `message` | `string` | Human-readable error message |
| `expected` | `string` | (type errors) Expected type |
| `received` | `string` | (type errors) Received type |
| `minimum` / `maximum` | `number` | (bound errors) Min/max bounds |
| `inclusive` | `boolean` | (bound errors) Whether bound is inclusive |
| `validation` | `string` | (string format errors) e.g., `'email'`, `'url'`, `'uuid'` |
| `options` | `string[]` | (enum errors) Valid enum values |

**`error.format()` example output** for a nested schema:
```ts
{
  consensus: { _errors: [] },
  contradictions: {
    _errors: ['Expected array, received string'],  // top-level array error
  },
  unique_insights: {
    _errors: [],
    0: { source: { _errors: ['Required'] } }       // nested field error
  },
  blind_spots: { _errors: [] },
}
```

**`error.flatten()` example output**:
```ts
{
  formErrors: [],
  fieldErrors: {
    'contradictions': ['Expected array, received string'],
    'unique_insights.0.source': ['Required'],
  }
}
```
