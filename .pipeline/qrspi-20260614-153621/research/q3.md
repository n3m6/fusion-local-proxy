## Findings for Q3

### Summary
Hono v4 provides three streaming helpers (`stream`, `streamText`, `streamSSE`) from `hono/streaming` that wrap the Web Streams API. `streamSSE` auto-configures SSE headers and provides a `writeSSE()` method on the `SSEStreamingApi` class; keep-alive/progress comments are emitted via the base `StreamingApi.write()` method. JSON responses use `c.json()`, and routes are organized via `app.route()` for sub-router mounting and `app.basePath()` for prefixing.

### Details

#### SSE Streaming API Surface
Hono v4 exports three streaming helpers from `hono/streaming`:
- **`stream(c, cb, onError?)`** — generic streaming; returns a `Response` with a `ReadableStream` body. The callback receives a `StreamingApi` instance.
- **`streamText(c, cb, onError?)`** — same as `stream` but adds `Content-Type: text/plain`, `Transfer-Encoding: chunked`, and `X-Content-Type-Options: nosniff` headers.
- **`streamSSE(c, cb, onError?)`** — SSE-specific streaming; sets `Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`, and `Transfer-Encoding: chunked`. The callback receives an `SSEStreamingApi` instance (which extends `StreamingApi`).

```ts
import { stream, streamText, streamSSE } from 'hono/streaming'
```
- `/tmp/hono-inspect/node_modules/hono/dist/types/helper/streaming/index.d.ts` (lines 1–6): exports `stream`, `streamSSE`, `SSEStreamingApi`, `streamText`, `SSEMessage`.
- Official docs at `https://hono.dev/docs/helpers/streaming` confirm the same API.

#### `SSEStreamingApi` and `SSEMessage`
- **`SSEMessage`** interface: `{ data: string | Promise<string>; event?: string; id?: string; retry?: number }`
- **`SSEStreamingApi.writeSSE(message)`**: serializes the `SSEMessage` into proper SSE wire format (`event:`, `data:`, `id:`, `retry:` lines separated by `\n`, terminated with `\n\n`). Multi-line `data` values are split on `\r\n|\r|\n` and each line prefixed with `data: `.
- `/tmp/hono-inspect/node_modules/hono/dist/types/helper/streaming/sse.d.ts` (lines 1–15): declares `SSEMessage` interface and `SSEStreamingApi` class with `writeSSE()`.
- `/tmp/hono-inspect/node_modules/hono/dist/helper/streaming/sse.js` (lines 6–30): implements `writeSSE()` serialization.

#### Keep-Alive / Progress Comment Emission
Hono v4 does **not** provide a dedicated method for SSE comment lines (keep-alive `: comment\n\n`). However, because `SSEStreamingApi` extends `StreamingApi`, the base `stream.write()` method can emit raw bytes — including SSE comment lines:
```ts
await stream.write(': keep-alive\n\n')
```
- `/tmp/hono-inspect/node_modules/hono/dist/utils/stream.js` (lines 1–61): `StreamingApi` class with `write(input: string | Uint8Array)`, `writeln(input)`, `sleep(ms)`, `close()`, `pipe(body)`, `onAbort(listener)`, `abort()`. Properties: `aborted: boolean`, `closed: boolean`.
- Comment pattern is standard SSE: a line starting with `:` followed by `\n\n` is ignored by SSE clients but keeps the connection alive.

#### Stream Lifecycle Management
All three streaming helpers share the same lifecycle pattern:
1. **Construction**: A `TransformStream` is created; its `writable` and `readable` ends are passed to the `StreamingApi` (or subclass) constructor.
2. **Execution**: The callback `cb(stream)` is invoked asynchronously. The stream auto-closes when the callback resolves (in a `finally` block).
3. **Error handling**: If the callback throws, the optional `onError(e, stream)` is called. For `streamSSE`, the error handler also writes an SSE `event: error` with `data: e.message`.
4. **Abort handling**: On old Bun, `c.req.raw.signal` abort triggers `stream.abort()`. The `onAbort()` method registers cleanup listeners. The `aborted` boolean flag is set.
5. **Back-pressure**: A `ReadableStream` wraps the reader, pulling chunks via `reader.read()` and closing when done. Client disconnect triggers the ReadableStream's `cancel()` which calls `stream.abort()`.
- `/tmp/hono-inspect/node_modules/hono/dist/utils/stream.js` (lines 1–61): full `StreamingApi` implementation.
- `/tmp/hono-inspect/node_modules/hono/dist/helper/streaming/sse.js` (lines 32–52): `run()` helper orchestrates try/catch/finally with auto-close and optional `onError`.
- `/tmp/hono-inspect/node_modules/hono/dist/helper/streaming/stream.js` (lines 1–33): `stream()` implementation showing the `TransformStream` + async IIFE pattern.

#### JSON Response Helpers
The Hono `Context` object provides:
- **`c.json(object, status?, headers?)`** — serializes a JSON-serializable value, sets `Content-Type: application/json`, and returns a `Response`. Accepts optional status code and headers.
- **`c.json(object, init?)`** — alternative overload accepting a `ResponseInit`-like object with `status`, `headers`, `statusText`.
- **`c.text(str, status?, headers?)`** — returns `Content-Type: text/plain`.
- **`c.html(str, status?, headers?)`** — returns `Content-Type: text/html`.
- **`c.body(data, status?, headers?)`** — generic body response (string, ArrayBuffer, Uint8Array, ReadableStream).
- **`c.status(code)`** — set HTTP status code.

#### Route Organization Patterns
Hono v4 supports these route organization conventions:

1. **Sub-router mounting with `app.route(path, subApp)`**: Create separate `Hono` instances for each resource/feature, then mount them onto a parent app. The parent's route path is prepended to all routes in the sub-app.
   ```ts
   const books = new Hono()
   books.get('/', (c) => c.json('list books'))
   books.get('/:id', (c) => c.json(`get ${c.req.param('id')}`))
   
   const app = new Hono()
   app.route('/books', books)  // mounts at /books and /books/:id
   ```
2. **Base path prefixing with `app.basePath(path)`**: Returns a new `Hono` instance where all routes are implicitly prefixed.
   ```ts
   const api = new Hono().basePath('/api')
   api.get('/book', (c) => c.text('List Books'))  // matches GET /api/book
   ```
3. **Chained method registration**: `.get().post().delete()` on the same path.
4. **HTTP method handlers**: `.get()`, `.post()`, `.put()`, `.delete()`, `.patch()`, `.options()`, `.all()`, `.on(method[], path[], handler)`.
5. **Middleware via `.use(path, middleware)`**: Applied in registration order; wraps route handlers.
6. **File-per-resource pattern**: Each `authors.ts` / `books.ts` exports its own `Hono` instance; `index.ts` imports and mounts them.

- `/tmp/hono-inspect/node_modules/hono/dist/types/hono-base.d.ts` (lines 70–86): `.route()` signature, `.basePath()` signature, `.mount()` signature.
- Official docs at `https://hono.dev/docs/api/routing` document Grouping, Base path, and Chained routes.
- Official docs at `https://hono.dev/docs/api/hono` lists all methods: `route`, `basePath`, `mount`, `on`, `use`, etc.

#### Node.js Serving
Hono itself is runtime-agnostic. To serve HTTP in Node.js, use `@hono/node-server`:
```ts
import { serve } from '@hono/node-server'
import { Hono } from 'hono'
const app = new Hono()
app.get('/', (c) => c.text('Hello Node.js!'))
serve(app)  // or serve({ fetch: app.fetch, port: 8787 })
```
- `/tmp/hono-inspect/node_modules/@hono/node-server/dist/index.d.mts` (lines 44–46): `serve(options: Options)` where `Options.fetch` is the Hono `app.fetch` callback.
- Official docs at `https://hono.dev/docs/getting-started/nodejs` confirm this pattern.
