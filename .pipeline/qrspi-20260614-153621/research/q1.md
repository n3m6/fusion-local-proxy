## Findings for Q1

### Summary
The OpenAI `/v1/chat/completions` endpoint accepts two required fields (`messages` and `model`) plus ~30 optional parameters spanning generation controls, tool calling, structured output, audio, streaming, and moderation. The non-streaming response is a `ChatCompletion` object with `choices` carrying `message` objects; the streaming response is a sequence of SSE `data:` lines carrying `ChatCompletionChunk` JSON payloads (with `object: "chat.completion.chunk"`), terminated by the literal `data: [DONE]` sentinel. No SSE `event:` type field is used for chat-completion streaming—the chunk type is identified solely by the `object` field in the JSON payload. #### 4. SSE Streaming Wire Format

##### 4a: Protocol
The streaming response uses the [Server-Sent Events protocol](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events/Using_server-sent_events#Event_stream_format) with `Content-Type: text/event-stream`. Each SSE "message" consists of one or more `data:` lines followed by an empty line (`\n\n`). The OpenAI implementation uses only the `data:` field; no `event:` field is emitted for chat-completion chunks.

##### 4b: Chunk Payload Format
Each chunk is a single `data:` line containing a JSON-serialized `ChatCompletionChunk`:

```
data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1694268190,"model":"gpt-4o","choices":[{"index":0,"delta":{"role":"assistant","content":""},"finish_reason":null}]}

data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1694268190,"model":"gpt-4o","choices":[{"index":0,"delta":{"content":"Hello"},"finish_reason":null}]}

data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1694268190,"model":"gpt-4o","choices":[{"index":0,"delta":{"content":" world"},"finish_reason":"stop"}]}
```

##### 4c: Chunk Sequencing
1. **First chunk**: `delta.role` is `"assistant"` (optionally with empty `content: ""`). `finish_reason: null`.
2. **Intermediate chunks**: `delta.content` contains incremental text. `finish_reason: null`.
3. **Final content chunk**: `delta.content` contains the last token(s). `finish_reason` is `"stop"` (or `"length"`, `"tool_calls"`, `"content_filter"`).
4. **Usage chunk** (only when `stream_options.include_usage: true`): `choices: []` (empty array), `usage` populated with full `CompletionUsage`. Arrives after the final content chunk and **before** `[DONE]`.
5. **Termination**: `data: [DONE]`

##### 4d: Termination Sentinel
```
data: [DONE]
```
- The literal string `[DONE]` (not JSON, no quotes, exactly this capitalization).
- Sent after the final chunk or usage chunk.
- Clients detect it via `sse.data.startsWith('[DONE]')` (confirmed in SDK at `/tmp/openai-sdk/package/core/streaming.js:33`).

##### 4e: SSE Comments (Keep-Alive / Progress)
Lines starting with `:` are SSE comments and are ignored by SSE parsers. The SDK's `SSEDecoder` class returns `null` for comment lines (`streaming.js:166`). Comments can be used for keep-alive pings and progress messages:
```
: keep-alive

: panel done

: judging
```
The project's planning documents (`goals.md:13`, AC-10) specify using such comments to cover non-streamed hops (panel and judge phases).

##### 4f: Error Handling
If an error occurs mid-stream, the SSE stream may contain a `data:` line with a JSON error object:
```
data: {"error": {"message": "...", "type": "...", "code": "...", "param": "..."}}
```
The SDK detects `data.error` and throws an `APIError` (`streaming.js:48`).

##### 4g: SSE Decoder Behavior (from SDK source)
From `/tmp/openai-sdk/package/core/streaming.js:152–178`:
- Empty lines trigger emission of the accumulated SSE event.
- Lines starting with `:` are ignored (SSE comment).
- Lines with `event:` set the `event` field.
- Lines with `data:` append to the `data` field (joined with `\n` for multi-line data).
- The resulting object: `{ event: string | null, data: string, raw: string[] }`.
- For chat completions, `event` is always `null`.
