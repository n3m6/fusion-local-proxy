## Findings for Q6

### Summary
Three distinct JSON config-schema approaches for multi-provider API gateways are surveyed: (1) a flat provider array with inline per-provider role assignment, (2) a two-level provider registry with separate named context groups, and (3) a model-centric routing schema where each configuration entry represents a specific model instance rather than a provider. Each pattern is observed in existing tools (OpenRouter, LiteLLM, generic Node.js/TypeScript backends) and analysed for complexity, extensibility, and discoverability.

### Details

#### Pattern 1 — Flat Provider Array with Inline Per-Provider Role Assignment

Each provider is a single flat object containing a type discriminator, a base URL, an API-key environment-variable reference, and a call-context assignment field (often named `role`, `mode`, or `context`). The configuration is a top-level array.

**Representative shape (conceptual, matching FR-9/FR-15 references):**
```json
{
  "providers": [
    {
      "type": "openai",
      "baseURL": "https://api.openai.com/v1",
      "apiKeyEnv": "OPENAI_API_KEY",
      "role": "chat"
    },
    {
      "type": "ollama",
      "baseURL": "http://localhost:11434",
      "apiKeyEnv": null,
      "role": "code-completion"
    }
  ]
}
```

- **Observed in**: OpenRouter client configurations (models listed with capability tags directly on each entry); simple multi-backend TypeScript projects that keep a single `providers` array in a JSON or YAML config file; early-stage Node.js proxy tools that start with a flat list before adding grouping.
- **Complexity**: Very low. A single `JSON.parse` + array iteration resolves all providers. Validation is a straightforward schema (e.g., Zod, JSON Schema) over a flat object.
- **Extensibility**: Moderate. Adding a new provider type requires only a new array element. However, if a single provider must serve multiple call contexts (e.g., the same OpenAI endpoint used for both chat and embeddings), the entry must be duplicated — each copy differing only in `role`. Adding optional provider-specific fields (e.g., `temperature`, `maxTokens`) inflates every object.
- **Discoverability**: High for simple topologies. Every provider and its purpose sits in one list. Discoverability degrades when the same provider appears multiple times; the reader must mentally merge duplicates to see the full provider inventory.

#### Pattern 2 — Provider Registry + Named Context Groups (Two-Level Separation)

Providers are defined exactly once in a dictionary (keyed by a logical name), and a separate `contexts` (or `groups`) section maps named call contexts to lists of provider keys. The application layer resolves a context name to its provider list, then looks up each provider by key.

**Representative shape:**
```json
{
  "providers": {
    "openai-main": {
      "type": "openai",
      "baseURL": "https://api.openai.com/v1",
      "apiKeyEnv": "OPENAI_API_KEY"
    },
    "ollama-local": {
      "type": "ollama",
      "baseURL": "http://localhost:11434",
      "apiKeyEnv": null
    }
  },
  "contexts": {
    "chat": ["openai-main"],
    "code-completion": ["openai-main", "ollama-local"],
    "embedding": ["openai-main"]
  }
}
```

- **Observed in**: LiteLLM proxy configurations (where `model_list` entries are registered and routing groups select subsets); TypeScript ORM/connection-pool patterns that separate *connection definitions* from *usage pools*; enterprise Node.js backends that use a `backends` + `routes` split in their config.
- **Complexity**: Medium. The config must be parsed in two stages (registry then group resolution), and the `ConfigPort` must expose methods that traverse the indirection (e.g., `getProvidersForContext(name)`). Reference integrity must be validated: every key in `contexts` must exist in `providers`.
- **Extensibility**: High. New call contexts can be added without touching provider definitions. A provider can participate in arbitrarily many contexts without duplication. Provider-specific settings stay local to the provider object. Adding optional per-context overrides (e.g., a different `baseURL` for the same provider in a different context) is possible by extending the context value from a string to an object with overrides.
- **Discoverability**: Medium. A human reader must cross-reference two sections to understand which providers serve a given context. The separation, however, makes each section independently coherent: the provider registry is a clean inventory; the context section is a clear routing table. Good for documentation but requires tooling for quick lookups in large configs.

#### Pattern 3 — Model-Centric Routing Configuration (Route/Model as Primary Entity)

Each configuration entry represents a specific *model instance* (provider + model name + parameters), not a generic provider. The type discriminator is embedded alongside model-specific routing parameters. Call-site assignment is either implicit (model naming convention or `mode` field) or handled by a separate routing layer.

**Representative shape (LiteLLM-inspired):**
```json
{
  "model_list": [
    {
      "model_name": "gpt-4-chat",
      "litellm_params": {
        "model": "gpt-4",
        "api_base": "https://api.openai.com/v1",
        "api_key": "os.environ/OPENAI_API_KEY",
        "temperature": 0.7
      },
      "model_info": {
        "mode": "chat",
        "max_tokens": 8192
      }
    },
    {
      "model_name": "claude-opus-code",
      "litellm_params": {
        "model": "claude-3-opus-20240229",
        "api_base": "https://api.anthropic.com",
        "api_key": "os.environ/ANTHROPIC_API_KEY"
      },
      "model_info": {
        "mode": "completion",
        "max_tokens": 4096
      }
    }
  ],
  "router_settings": {
    "routing_strategy": "usage-based"
  }
}
```

- **Observed in**: LiteLLM proxy (the `model_list` with `litellm_params` and `model_info` is its canonical config format); OpenAI-compatible gateways that map many upstream model names to a unified API surface; TypeScript projects that treat each “capability” as a configured service rather than a generic provider.
- **Complexity**: High. Each model entry is verbose and contains nested parameter objects. The config schema must accommodate provider-specific parameter shapes beneath a common envelope (e.g., `litellm_params`). Parsing requires a model-factory pattern that dispatches on the provider type *within each entry*. The `ConfigPort` must expose model-level query methods.
- **Extensibility**: Very high. Per-model overrides (rate limits, retry policies, prompt templates, token limits) are first-class. Adding a new provider means adding a new model entry — no change to the schema. Multiple models from the same provider are naturally independent entries. The routing layer can implement load-balancing, fallback, and cost-optimisation strategies across entries.
- **Discoverability**: High at the model level (every capability is a named, documented entry), but provider-level discoverability is low — understanding which providers are configured requires scanning all entries and extracting unique `api_base` values. The config file can become very large in production deployments.

#### Cross-Cutting Observations

The question framing references a `ConfigPort` interface (NFR-6) that insulates the application from the config file format, and a `JsonFileConfigAdapter` (AC-6) that satisfies it. All three patterns can be backed by such a port/adapter arrangement; the choice of schema does not dictate the interface shape, but it does influence which query methods the port must expose:

- **Pattern 1** needs `getProviders()` → array, and callers filter by `role` themselves, or a convenience `getProvidersByRole(role)`.
- **Pattern 2** needs `getProvidersForContext(context)` → array, plus `getProvider(key)` for direct lookup.
- **Pattern 3** needs `getModel(modelName)` → model config, plus `listModels(filter?)` for discovery.

The `ChatAdapterFactory` selection by `provider.type` (FR-10) is orthogonal to the grouping strategy — it operates on individual provider/model objects regardless of how they are organised in the config file.
